# ARCHITECTURE

## Слои приложения

- `src/data` хранит карты, главы, сцены, события и mapping изображений карт;
- `src/domain` содержит player state, progress helpers, storage helpers и journal snapshot;
- `src/app.ts` собирает экраны, управляет переходами и рендерит HTML;
- `src/style.css` отвечает за mobile-first layout и читаемую подачу;
- `public/assets/cards/` содержит локальные миниатюры карт Таро.

## Ключевые доменные сущности

- `TarotCard` - базовое описание карты;
- `StoryChapter` - большая глава пути;
- `DialogueScene` - экран сцены text quest;
- `DialogueChoice` - действие игрока;
- `PlayerState` - прогресс, inventory state и сохранённый путь;
- `JournalSnapshot` - агрегированный вид следов пути для `Дневника Шута`.

Для feedback слоя `PlayerState` также хранит сведения о последней полученной карте и о том, была ли она новой для Карты Шута, чтобы result screen мог различать `Получено`, `Применено` и `Помощник пути`.

## Экранная архитектура

- `home` - главный экран с CTA и вторичным доступом к дневнику;
- `scene` - чистая игровая сцена без длинной страницы, собранная как classic quest layout;
- `result` - отдельный экран обратной связи после выбора, а не лог;
- `journey` - переходный экран для совместимости с текущим flow;
- `journal` - отдельный экран коллекции карты и пути;
- `completion` - финальный экран после Мира.

## Story flow

Смысл сцены собирается из нескольких слоёв:

- базовая карта;
- контекст главы или дорожного события;
- ориентация;
- прямой вопрос игроку;
- точечные overrides под выбор;
- inventory state, если карта была получена ранее.

## Карты и изображения

- в коде есть явный mapping `cardId -> image`;
- изображения не скачиваются во время работы приложения;
- fallback нужен для редких случаев отсутствия изображения;
- источник ассетов задокументирован в `docs/ASSETS.md`.

## Ограничения

- без backend и внешних API;
- без авторизации и серверного хранения;
- без тяжёлой RPG-экономики;
- без генерации случайных раскладов как основной механики;
- с обязательной проверкой мобильной читаемости и отсутствия горизонтальной прокрутки.
