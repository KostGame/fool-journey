# Техническое задание для Codex: игра «Путь Шута»

## 1. Краткое описание

Нужно разработать браузерную игру для изучения Таро.

Рабочее название:

```text
Путь Шута
```

Суть: игрок проходит путешествие от Шута к Миру, знакомится с картами как с персонажами, решает лёгкие сюжетные ситуации, учится видеть значения карт в контексте вопроса, позиции, прямого или перевёрнутого положения и соседних карт.

Это не должна быть обычная интеллектуальная викторина «угадай карту по описанию». Викторины допустимы как один из режимов, но ядро игры должно быть сюжетным, интерактивным и спокойным.

Цель первой версии: создать техническую основу, игровую концепцию, базовый прототип кампании и GitHub workflow для дальнейших SU-задач.

---

## 2. Основные принципы проекта

1. Приложение должно запускаться в браузере.
2. Деплой через GitHub Pages.
3. CI и деплой через GitHub Actions.
4. Без backend.
5. Без внешних API.
6. Без обязательного интернета после загрузки страницы.
7. Состояние игрока хранить локально, через `localStorage` или IndexedDB.
8. UI на русском языке.
9. Mobile-first, основной сценарий использования: смартфон.
10. Визуальный стиль: мистический, тёплый, немного игровой, но не перегруженный.
11. Игра должна быть весёлой, не напряжной, без ощущения экзамена.
12. Трактовки должны помогать учиться, но не выдавать мистические утверждения как гарантированный прогноз.
13. Все будущие задачи оформлять через GitHub Issues и отдельные SU-файлы в репозитории.
14. Codex после каждой задачи должен оставлять отчёт в папке `reports/`.

## 4. Продуктовая концепция

### 4.1. Главная метафора

Игрок становится Шутом, который идёт по дороге через 22 старших аркана и постепенно открывает младшие арканы как инструменты повседневных ситуаций.

Каждая карта не просто термин, а персонаж, место, испытание или настроение.

Пример:

- Маг: «собери намерение и выбери инструмент».
- Жрица: «не всё видно сразу, найди скрытый смысл».
- Императрица: «что нужно вырастить, поддержать, оживить».
- Башня: «что рушится и почему это может быть освобождением».
- Мир: «как собрать опыт в целую картину».

### 4.2. Игровой цикл

Базовый цикл одной игровой сцены:

1. Игрок попадает в короткую ситуацию.
2. Игра показывает одну или несколько карт.
3. Игрок выбирает, как прочитать карту в этом контексте.
4. Игра не просто говорит «правильно/неправильно», а объясняет, почему выбранный смысл подходит или не подходит.
5. Игрок получает мягкий прогресс: опыт, понимание карты, отметку в дневнике.
6. Открывается следующая сцена, режим или усложнение.

### 4.3. Почему это не просто викторина

В игре должны быть разные типы взаимодействий:

- выбор трактовки по жизненной ситуации;
- сборка ответа из смысловых фрагментов;
- сопоставление карты с ролью в раскладе;
- выбор уточняющей карты;
- сравнение двух похожих карт;
- мини-диалоги с архетипами;
- дневник наблюдений;
- тренировочные расклады.

## 5. Основные режимы игры

### 5.1. Режим «Путь Шута»

Главный сюжетный режим.

Игрок проходит главы по старшим арканам. Каждая глава открывает одну карту и даёт 2 или 3 коротких интерактивных сцены.

Структура главы:

1. Вступительная сцена.
2. Знакомство с архетипом карты.
3. Практическая ситуация.
4. Мини-расклад.
5. Итоговое правило карты, коротко и понятно.

Пример главы «Маг»:

- ситуация: «У тебя есть идея, но ты всё откладываешь»;
- карта: Маг;
- вопрос: «Что здесь главное?»;
- варианты: «начать с доступного инструмента», «ждать идеального момента», «попросить знак»;
- после выбора игра объясняет: Маг про намерение, волю, действие и использование ресурсов.

### 5.2. Режим «Живой расклад»

Тренировка чтения карт в контексте.

Игрок выбирает тип вопроса:

- карта дня;
- совет;
- что происходит;
- выбор между вариантами;
- что будет, если я сделаю это;
- отношения;
- работа и дело;
- внутренняя причина;
- предупреждение;
- следующий шаг.

Игра предлагает расклад на 1, 3 или 3+1 карты.

Для каждой позиции показывается не только значение карты, но и значение именно на этом месте.

Пример для расклада 3+1:

```text
1. Суть ситуации
2. Что помогает
3. Что мешает
4. Совет
```

### 5.3. Режим «Аркана-диалоги»

Карты выступают как персонажи.

Игрок видит реплику персонажа и должен понять, какая карта так говорит, или выбрать, что этот архетип советует.

Пример:

```text
«Ты уже знаешь ответ, но боишься услышать тишину»
```

Возможные карты:

- Жрица;
- Маг;
- Колесница.

Игра объясняет, почему Жрица подходит лучше.

### 5.4. Режим «Собери трактовку»

Игрок собирает короткий ответ из смысловых блоков.

Например, карта: Двойка Пентаклей, позиция: «что мешает».

Доступные фрагменты:

- «ты пытаешься удержать несколько процессов сразу»;
- «важна гибкость»;
- «нужно выбрать один главный ритм»;
- «всё уже разрушено окончательно».

Игрок выбирает 2 или 3 фрагмента, а игра показывает собранную трактовку и объясняет, какие блоки были точными.

### 5.5. Режим «Похожие, но разные»

Тренировка различения карт, которые легко спутать.

Примеры пар:

- Маг и Жонглёрская энергия Двойки Пентаклей;
- Императрица и Королева Пентаклей;
- Башня и Смерть;
- Луна и Дьявол;
- Звезда и Умеренность;
- Шестёрка Кубков и Ностальгия Луны;
- Колесница и Рыцарь Жезлов.

Игрок получает ситуацию и должен выбрать, какая карта точнее.

### 5.6. Режим «Карта дня»

Лёгкий ежедневный режим.

Игрок вытягивает карту дня и получает:

- короткий смысл;
- настроение дня;
- вопрос к себе;
- маленькое действие;
- возможность отметить вечером, как карта проявилась.

История хранится локально.

### 5.7. Режим «Свободная практика»

Игрок может выбрать любую карту, позицию и тип вопроса.

Это режим справочника и тренажёра одновременно.

## 6. Прогрессия и уровни

### 6.1. Общий уровень игрока

Игрок получает опыт за:

- прохождение сцен;
- ежедневную карту;
- чтение раскладов;
- корректные выборы;
- осознанные исправления после ошибки;
- возвращение к сложным картам.

Важно: не делать жёсткое наказание за ошибки. Лучше давать мягкую обратную связь.

Пример уровней:

```text
1. Шут у ворот
2. Искатель знаков
3. Ученик арканов
4. Толкователь ситуаций
5. Проводник расклада
6. Мастер контекста
```

### 6.2. Уровень владения отдельной картой

Для каждой карты хранить прогресс:

```text
0. Не открыта
1. Встречена
2. Узнана
3. Понята в простом контексте
4. Понята в раскладе
5. Использована в самостоятельной трактовке
```

### 6.3. Открытие контента

Рекомендуемое усложнение:

1. Сначала 22 старших аркана.
2. Потом четыре масти как четыре стихии опыта.
3. Потом числа младших арканов.
4. Потом придворные карты.
5. Потом перевёрнутые карты.
6. Потом сочетания карт.
7. Потом сложные типы вопросов.

### 6.4. Достижения

Примеры достижений:

- «Первый шаг», пройти главу Шута;
- «Не всё так страшно», понять Башню и Смерть в разных контекстах;
- «Собрал масть», открыть все карты одной масти;
- «Три карты, один смысл», пройти первый расклад 3 карты;
- «Дневник ожил», отметить 7 карт дня;
- «Без зубрёжки», 10 раз выбрать смысл через ситуацию, а не через название.

## 7. Учебная модель Таро

### 7.1. Структура карты

Для каждой карты нужно хранить:

```ts
id: string;
name: string;
group: 'major' | 'wands' | 'cups' | 'swords' | 'pentacles';
rank?: string;
keywords: string[];
lightMeaning: string;
shadowMeaning: string;
advice: string;
warning: string;
dailyMeaning: string;
questionToSelf: string;
storyRole: string;
symbols: string[];
commonMistakes: string[];
```

### 7.2. Контекстная трактовка

Трактовка должна зависеть от:

- карты;
- типа вопроса;
- позиции в раскладе;
- прямого или перевёрнутого положения;
- соседних карт, в будущих версиях;
- уровня игрока.

Для первой версии достаточно заложить архитектуру и заполнить часть данных, не обязательно сразу все 78 карт на максимальной глубине.

### 7.3. Типы вопросов

Минимальный набор типов вопросов:

```text
card-of-day
advice
situation
choice
if-i-do-this
relationship
work
inner-cause
warning
next-step
```

### 7.4. Позиции раскладов

Минимальные расклады:

```text
1 карта:
- Ответ / совет

3 карты:
- Суть
- Что помогает
- Что мешает

3 + совет:
- Суть
- Скрытый фактор
- Возможное развитие
- Совет

Выбор:
- Вариант А
- Вариант Б
- Что важно учесть
```

## 8. UX и визуальный стиль

### 8.1. Главный экран

На главном экране должны быть:

- название «Путь Шута»;
- текущий уровень игрока;
- прогресс путешествия;
- большая кнопка «Продолжить путь»;
- дополнительные режимы:
  - Живой расклад;
  - Карта дня;
  - Аркана-диалоги;
  - Собери трактовку;
  - Справочник.

### 8.2. Мобильный интерфейс

Основной экран должен удобно работать на ширине 360–430 px.

Требования:

- без горизонтальной прокрутки;
- крупные области нажатия;
- короткие тексты;
- карточки и сцены должны помещаться без лишней перегрузки;
- длинные объяснения раскрывать по кнопке «почему так»;
- не делать интерфейс похожим на школьный тест.

### 8.3. Тональность текстов

Стиль текстов:

- доброжелательный;
- немного ироничный;
- без пафоса;
- без запугивания;
- без утверждений вида «точно случится»;
- больше в формате «эта карта может подсвечивать...».

## 10. GitHub workflow

### 10.1. Общий процесс

1. Пользователь и ChatGPT формируют задачу.
2. Задача кладётся в GitHub Issue.
3. Для крупной задачи дополнительно создаётся SU-файл в `tasks/SU/`.
4. Codex создаёт ветку.
5. Codex делает изменения.
6. Codex запускает проверки.
7. Codex открывает Draft PR.
8. Codex кладёт отчёт в `reports/`.
9. Codex добавляет комментарий в Issue со ссылками на PR и report.
10. Пользователь присылает отчёт в чат.
11. ChatGPT анализирует отчёт и готовит следующую SU-задачу.

### 10.2. Формат SU-файлов

Папка:

```text
tasks/SU/
```

Именование:

```text
SU-000-bootstrap.md
SU-001-core-game-loop.md
SU-002-card-data-model.md
SU-003-story-campaign-majors.md
```

Шаблон SU:

```md
# SU-XXX: Название

## Цель

## Контекст

## Что сделать

## Что не делать

## Файлы и зоны изменений

## UX-требования

## Тесты и проверки

## Критерии готовности

## Формат отчёта
```

### 10.3. Формат отчётов Codex

Папка:

```text
reports/
```

Именование:

```text
reports/PR-000-bootstrap.md
reports/PR-001-core-game-loop.md
```

Отчёт должен содержать:

```md
# Отчёт Codex по PR-XXX

## 1. Кратко

## 2. Ветка и PR

## 3. Что реализовано

## 4. Что изменено по файлам

## 5. Проверки

## 6. Что не сделано

## 7. Риски и ограничения

## 8. Что проверить вручную

## 9. Можно ли переводить PR в Ready for review
```

## 11. Roadmap разработки

### PR-000: Bootstrap, Pages, документация и игровой каркас

Цель: создать репозиторий, инфраструктуру, документацию и первый кликабельный каркас.

Состав:

- Vite + TypeScript;
- GitHub Actions CI;
- GitHub Pages deploy;
- README, AGENTS, docs;
- базовый mobile-first UI;
- главный экран;
- placeholder режимов;
- начальная модель состояния игрока;
- 3–5 тестов для storage/progress/random;
- отчёт в `reports/`.

### PR-001: Core game loop

Цель: сделать первый полноценный игровой цикл.

Состав:

- сцена;
- карта;
- выбор действия;
- обратная связь;
- начисление опыта;
- сохранение прогресса;
- 3–5 сцен для Шута, Мага и Жрицы.

### PR-002: Карты и учебная модель

Цель: заложить структуру данных по картам.

Состав:

- модель карты;
- базовые данные 22 старших арканов;
- ключевые слова;
- совет, предупреждение, вопрос к себе;
- экран справочника.

### PR-003: Режим «Живой расклад»

Цель: научить игрока читать карту в позиции.

Состав:

- типы вопросов;
- расклады 1, 3, 3+1;
- контекстная трактовка;
- объяснение позиции;
- сохранение истории раскладов локально.

### PR-004: Карта дня и дневник

Цель: ежедневная практика.

Состав:

- вытягивание карты дня;
- история дней;
- вечерняя отметка;
- вопрос к себе;
- простая статистика.

### PR-005: Младшие арканы и масти

Цель: добавить младшие арканы через систему масть + число + ранг.

Состав:

- 4 масти;
- числа 1–10;
- придворные карты;
- объяснение формулы младших арканов;
- тренировки по мастям и числам.

### PR-006: Перевёрнутые карты

Цель: добавить перевёрнутые карты как усложнение, не по умолчанию.

Состав:

- настройка включения;
- отдельный учебный блок;
- мягкие трактовки перевёрнутых карт;
- предупреждение, что перевёрнутость можно не использовать.

### PR-007: Сочетания карт

Цель: учить читать не отдельные карты, а связи.

Состав:

- пары карт;
- соседние карты;
- усиление, конфликт, развитие;
- режим «Похожие, но разные».
